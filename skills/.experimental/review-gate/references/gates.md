# Gate catalog

Admission rule: a gate enters this catalog only from a recorded case ([cases/](../../../../cases)) or a confirmed external-review miss. A gate is marked **promoted** when it recurs across two or more independent cases or a maintainer-confirmed miss validates it. Everything else is **candidate**.

Each entry carries: what triggers it, the binary check or pass question, and provenance.

## Deterministic gates

## Lenses

One focused pass per triggered lens. Findings from a lens are candidates until step 4 (verify) confirms them.

### Inverse regression surface (promoted)

- **Trigger:** the diff replaces a matcher, parser, or lookup's data source (syntactic → semantic, one API → another).
- **Pass question:** the change has two regression surfaces, not one. Beyond "what does the new source see that the old one missed?", ask the inverse: "what did the old source accept that the new one erases?" Values whose meaning is absence from the new source are structurally invisible to a test matrix built from the first question alone.
- **Provenance:** [1552](../../../../cases/agent-browser/1552-getbyrole-implicit-roles-regression.md) (third lesson: presentational roles are exactly the values a semantic source omits); converges with the new-domain matrix lens across two repos.

### New-domain matrix (promoted)

- **Trigger:** the diff widens a trigger condition or relaxes a validator (a guard accepts more inputs than before).
- **Pass question:** the verification matrix is regenerated from the new input domain, not inherited from the bug report. Enumerate the input classes the code now reaches, the previously-impossible values flowing into every downstream consumer, and exercise one of each. Helpers the code reuses carry gaps that were benign only because they were unreachable. Derive cells from the input domain the composed helpers accept, never from the argument positions the diff's own guard inspects: the guard is the artifact under test, and cells derived from its shape re-encode its blind spot (every guard cell needs a twin for each transformation the composed helpers tolerate, e.g. runner-wrapped forms). Enumerate each downstream consumer to its substrate, not its semantics: a value valid at the logical layer (a 253-char hostname is valid DNS) can be invalid at the constraint the consumer writes to (a cert filename over the filesystem `NAME_MAX`), so the length or format axis runs once per consumer substrate, not once for the domain (portless #365 cert-path miss, ctate-confirmed 2026-07-20). When the fix adds its own guard (a detector, matcher, or tokenizer), that guard is itself under test: give it a cell per adversarial form of what it detects (shell operators without surrounding spaces `dev&&node`, newline separators, `--flag=value` forms), because a detector's cells silently inherit its own tokenization assumptions (portless #366-B glued-operator miss, ctate-confirmed 2026-07-20). A compound/script detector's adversarial-form cells also include quoted shell metacharacters and redirections the shell still honors (`"foo & bar"`, `cmd > out.log`, `<` heredoc-ish forms), not only bare operators; and when the diff injects flags, the matrix must include the **partial-injection** cell where one target flag is already present and its sibling absent — injecting a missing `--host` must not be gated on `--port` also being missing (each injected flag is independent). (portless #366 round 3, ctate-confirmed 2026-07-22.) When the diff adds a heuristic probe whose verdict is inferred from non-response, enumerate the blocked-but-alive states from the product's own feature list, not from the bug report: every feature that can pause or block the probed component (modal dialogs, debugger pauses, attach-time pauses) is an input class, and the feature list is sitting in the repo's own command surface.
- **Provenance:** portless [365](../../../../cases/portless/365-risky-suffix-overgeneralization.md) (multi-segment TLDs: both bugs lived in matrix cells outside the reported case) and [366](../../../../cases/portless/366-guard-derived-matrix.md) (script indirection round 1: build scripts, compound scripts, `--flag=value` forms; round 2, confirmed external miss: `"dev": "bunx vite build"` bypassed the `rawScript[1]` build guard because the matrix's build cells were derived from the guard's shape while `findFrameworkBasename` skips runner wrappers); same mechanism as the inverse-regression-surface lens. Plus agent-browser [1532](../../../../cases/agent-browser/1532-discarded-tab-revival.md) round 2 (maintainer-confirmed, reproduced in Chrome): the liveness probe's matrix enumerated tab lifecycle states (responsive, slow, discarded, dead) but missed renderer-blocked-but-alive, an open `confirm()`/`prompt()` pauses the main thread and reads as discarded; the class was discoverable from the repo's own `dialog accept/dismiss` commands.

### Resolution-rule consistency across consumers (candidate)

- **Trigger:** the diff changes how a hostname, route, or key is resolved from user input when entries can overlap (longest-match, precedence order, prefix/suffix rules).
- **Pass question:** every code path that reconstructs the same value from the same input applies the identical resolution rule, not only the primary resolver. Error builders, 404/suggestion text, logging, and cleanup routines frequently re-split the input with a naive rule and produce a plausible-but-wrong answer. Enumerate consumers of the parsed input (`gate.sh callers <parse/resolve fn>`), and for an overlapping-entry case (`example.com` and `dev.example.com` both registered) force the suggestion/error path and read the exact value it emits.
- **Provenance:** portless [365](../../../../cases/portless/365-risky-suffix-overgeneralization.md) round 3, ctate-confirmed 2026-07-22: the resolver used longest-match for overlapping TLDs, but the 404 suggestion builder did not, so `missing.dev.example.com` suggested `portless missing.dev your-command`, registering the wrong hostname set.

### Emission channel and one-shot latch reachability (candidate)

- **Trigger:** the diff adds a user-facing warning/error, OR guards a "warn/do once" action behind a latch, flag, or first-run check, in a system with a background daemon/process distinct from the CLI the user is attached to.
- **Pass question:** two properties, both forced. (1) The message is emitted on the process the user actually watches. A warning written by the daemon lands only in the proxy/daemon log; the user running the CLI never sees it. Trace which process executes the emission and confirm it reaches the attached stdio. (2) A one-shot latch is not consumed by an internal call before the user-triggered path runs: an initial empty/warmup pass (e.g. the first hosts sync with nothing to write) must not flip the latch and silence the real event. Force the warm-up-then-real sequence and assert the user still sees it.
- **Provenance:** portless [367](../../../../cases/portless/367-daemon-emit-latch.md), ctate-confirmed 2026-07-22: the hosts-sync-failure warning was emitted by the daemon (proxy log only) and the initial empty sync consumed the warning latch, so route registration stayed silent for the user.

### Shim hermeticity (candidate)

- **Trigger:** a test in the diff stubs or shims an external binary (`bun`, `node`, `npx`, a package runner) via a fake file, PATH entry, or mock.
- **Pass question:** the shim is guaranteed to win over a really-installed binary of the same name. A test that prepends a fake `bun` to PATH but runs on a machine where the real `bun` resolves first (different lookup, cached path, absolute invocation) silently exercises the real tool and passes for the wrong reason. Verify the shim's precedence explicitly (isolated PATH, absolute mock path, or asserting the shim ran), and treat a green test whose shim could be shadowed as unwritten.
- **Provenance:** portless #366 round 3, ctate-confirmed 2026-07-22: the new integration tests failed when Bun was installed beside Node because the real binary shadowed the test shim.

### Deliberate-default check (candidate)

- **Trigger:** the diff changes existing default behavior (removes or alters what the code did before, rather than adding a new path).
- **Pass question:** the current behavior is proven accidental, not deliberate. Search the tests that assert it and the recent changelog/release notes that shipped it; behavior with tests asserting it (worst case: same-day tests) is a feature, and the diff must become opt-in or a design conversation instead of a silent default flip. The author's own reasoning, live validation, and written rationale do not substitute — only the target repo's suite knows what is intentional.
- **Provenance:** [native 2026-07-17](../../../../cases/native/2026-07-17-arrow-default-was-todays-feature.md) (arrow-forwarding patch reverted the v0.5.2 edit-derivation feature shipped hours earlier; the repo's own gate caught it pre-push).

### Fresh-seam scan (candidate)

- **Trigger:** the diff patches a runtime or framework to expose a signal to application code (a new channel, callback, or forwarded event).
- **Pass question:** the latest release notes and changelog of the target repo have been scanned for a seam that already carries the signal. The capability often just shipped through another door, and the patch is then both conflicting and unnecessary.
- **Provenance:** [native 2026-07-17](../../../../cases/native/2026-07-17-arrow-default-was-todays-feature.md) (the hours-old edit-derivation seam already delivered the arrow keys as `move_caret` edits; the runtime patch was withdrawn and the feature built pure-userland).

### Reference-implementation oracle (candidate)

- **Trigger:** the diff reimplements behavior that has a reference implementation or a written spec (a Playwright semantic, a W3C contract, an RFC).
- **Pass question:** the reference's full contract is enumerated and each behavior is checked, not only the cases the bug report cited. Borrowing one semantic from the reference while ignoring its neighbors is how the long tail ships broken. **Where the contract is enumerable, run it as a forced checklist, not a prose reminder** — for agent-browser getByRole, that checklist is [getrole-contract.md](getrole-contract.md), applied to any role-matching diff.
- **Provenance:** agent-browser #1552 round-2 misses (role-synonym normalization, ordered fallback role lists); every one already solved by the reference implementation the fix cited. **Blind-replication A/B (2026-07-20):** the prose oracle in conventions already named both "AX-role to ARIA-name normalization" and "scope `--exact` to role accessible names", yet a blind codex reviewer with the prose caught 0/2; the same reviewer with the same facts as a forced line-item checklist ([getrole-contract.md](getrole-contract.md)) caught 2/2 (F2 `directory`→`list`, F3 `--exact` doc scope). The prose→checklist format is the isolated variable. A norm left as prose runs as judgment, and judgment sampled past both. Report: `05_Areas/vercel/reviewgate-validation-2026-07-20/`.

### New-failure-outcome propagation (candidate)

- **Trigger:** the diff converts a path that previously could only succeed or hang (or crash) into one that returns an error, or otherwise adds a new outcome class to an existing call.
- **Pass question:** every caller of the now-failable function is walked for state mutated before the call: clears, commits, and side effects that were benign while failure was impossible are now executed on a path that returns to the user. The callers are typically NOT in the diff, which is exactly why they break. Enumerate them mechanically with `gate.sh callers <fn>` (or a radius Impact Map where available); never from memory. Verification runs at the caller's layer through the real dispatch path, asserting the full user-visible state after the forced failure, not the single invariant the fix targeted.
- **Provenance:** agent-browser [1532](../../../../cases/agent-browser/1532-discarded-tab-revival.md) round 2 (maintainer-confirmed): `handle_tab_switch` cleared `ref_map`, `iframe_sessions`, and `active_frame_id` before calling `tab_switch`; the clears were benign while the switch could only succeed or hang forever, and the fix's new fail-fast path returned into a handler that had already wiped the old tab's refs. The gate's own mock test asserted only `active_page_index` at the `BrowserManager` layer, one layer below where the user-visible state lives.

### Flag-propagation dispatch sweep (candidate)

- **Trigger:** the diff adds a flag, mode, or config bit that alters the behavior of an existing subsystem reachable from more than one entry point.
- **Pass question:** the flag is traced through every dispatch path that reaches the subsystem, enumerated from the code, not from the diff: fresh launch, auto-launch, connect/attach, re-attach/restore, wrapper frontends (batch runners, MCP tool schemas, preliminary command envelopes), and event handlers. Per path: is the flag applied, at the right time, to the right object? Wrappers that rebuild command envelopes independently are where the flag silently drops; fingerprint/reuse functions that decide daemon reuse are where a stale un-flagged instance survives; a flag applied after the state it governs was already selected is a miss even when the path nominally supports it.
- **Provenance:** agent-browser [1589](../../../../cases/agent-browser/1589-pin-tab-dispatch-paths.md) (maintainer-confirmed round-1 misses: `--pin-tab` unset on local launch paths, applied without binding on a running daemon, dropped by `run_batch` and the CDP launch envelope, excluded from `daemon_config_fingerprint`). Blind replication on the same diff: the generic catalog reached the silent-recovery and batch findings but not the dispatch-path ones; this enumeration is what reaches them agnostically.

### Error-path forcing (promoted)

- **Trigger:** the diff adds any branch, guard, wrapper, or "this can't happen" fallback.
- **Pass question:** each new branch is its own claim, validated by forcing it: the worst case is observed, not reasoned about. Check ordering too: validation that runs after expensive setup produces errors that name the wrong fault. Wrapped errors preserve the guidance the unwrapped path carried.
- **Provenance:** [1553](../../../../cases/agent-browser/1553-error-detail-and-help-drift.md) (primary lesson: the "impossible" fallback was a silent client-side timeout) + round-2 misses (validation ordering, wrapped-error guidance).

### Non-destructive recovery (candidate)

- **Trigger:** the diff adds a recovery action driven by a heuristic detector (no reliable signal exists).
- **Pass question:** the recovery is harmless on a false positive. Making the false positive cost nothing is stronger than tuning the detector to misfire less often.
- **Provenance:** [1532](../../../../cases/agent-browser/1532-discarded-tab-revival.md) (activation recovers a dead renderer but only focuses a live one).

### Cancellation and timeout hygiene (candidate)

- **Trigger:** the diff adds an async operation, a timeout, or a cancellable request.
- **Pass question:** every new wait is bounded below the ceiling it could otherwise ride; a cancelled operation leaves no pending state behind (verified by a leak test with teeth: disabling the guard must make it fail).
- **Provenance:** [1532](../../../../cases/agent-browser/1532-discarded-tab-revival.md) (unbounded reload; cancelled probe leaked its pending entry).

### Boundary pipeline trace (candidate)

- **Trigger:** the diff extends a feature across a new boundary (a frame, a shadow root, a process, a protocol).
- **Pass question:** every pipeline stage crosses the boundary: discovery, resolution, action, cleanup. Discovery alone is useless when a later stage still stops at the old boundary.
- **Provenance:** [1266](../../../../cases/agent-browser/1266-shadow-dom-locators.md).

### Substrate verification (promoted)

- **Trigger:** the change's success is asserted from a command's own output, or verified against a long-lived process.
- **Pass question:** the effect is observed on the substrate (DOM, filesystem, serialized artifact, process identity), not on the command's report of itself. When a daemon or server may have answered, the process that responded is identified before either the broken or the fixed result is trusted.
- **Provenance:** [1105](../../../../cases/agent-browser/1105-select-silent-success.md), [1204](../../../../cases/agent-browser/1204-har-response-bodies.md), [shared-daemon](../../../../cases/agent-browser/shared-daemon-cross-worktree-contamination.md).

### Docs-behavior parity (candidate)

- **Trigger:** the diff changes observable behavior (semantics, defaults, matching rules, accepted inputs, output shape), not only internals.
- **Pass question:** first enumerate the behavior deltas the diff introduces, establishing each empirically against the built artifact; then, for each delta, walk the project's surface list (CLI help, MCP descriptions, README, web docs, skill files) and check that each surface's prose states the new behavior precisely, including its scope: what the behavior applies to and what it deliberately does not. The deterministic surface sweep proves the files were touched; this lens proves the words agree with the binary. An assertive sentence that misstates the behavior (broader, narrower, wrong, or ambiguous in scope) is always a finding; silence is a finding only on the surface that carries the full contract for that behavior. A terse command listing need not restate semantics. **Walk passages, not files.** A single behavior is often described in several places within one file (a feature section and a canonical "how it works" section, a help block and a config table) and in the CLI `--help` prose. A file being in the diff for one passage does not exempt its other passages: enumerate every passage that describes the feature (grep the keyword across the whole repo, then within each hit file re-read the lines the diff did not touch) and check each. The deterministic sweep is file-level and passes any touched file; this lens is where the intra-file stale sibling gets caught. The goal is to find a passage that is now stale or contradictory, not to repeat the new sentence in every passage: state the behavior once per file at its canonical passage (the dedicated "how it works" line), and leave topical or terse siblings that do not claim completeness alone. Duplicating the same caveat across several passages of one file is over-documentation, noise for the agents and humans who read the whole file. **Inverse direction:** widening a domain also falsifies pre-existing claims about the widened surface, on lines the diff never touches. For each behavior the diff enables, grep the docs for existing assertions about that surface (network reach, platform support, "works across X") and check they still hold under the widened domain and against invariants that landed after the branch was cut. A sentence true when written becomes a lie the moment a validator relaxes or a hardening binds narrower (portless #365, ctate-confirmed 2026-07-20: "custom TLDs work across LAN/tailnet" while `--lan` forces `.local` and normal mode binds loopback-only). **Advertised-capability direction:** for every capability a surface advertises (an action or flag in CLI `--help`, MCP schema, README, or docs), confirm the code path actually delivers it end to end, not just that a doc mentions it. When a diff widens what a surface advertises (adds an action to a locator's usage, a flag to a command), trace each newly advertised path to dispatch and check the value/argument survives. Advertising a capability the code drops is a docs-behavior mismatch caught from the docs side (agent-browser #1553 round 3, ctate-confirmed: `fill` advertised for the text/alt/title locators while their command arms dropped the fill value before dispatch).
- **Provenance:** agent-browser #1552, both review rounds (round 1: help and MCP still described the old `--exact` semantics; round 2: docs did not scope case-insensitive matching to role accessible names). Two maintainer-confirmed misses; the only answer-key item invisible to the catalog in blind-replication round 002. Plus portless #363 (maintainer-confirmed): the terse-listing exemption was misapplied to a CLI help section that carries behavior prose — the exemption covers command/flag listings only; a bolded feature section whose sibling in README gained a behavior sentence is a contract-carrying surface, and silence there is a finding.

### Demonstrative example (candidate)

- **Trigger:** the diff adds or edits a documentation example meant to demonstrate a capability.
- **Pass question:** the example actually requires the capability: it would fail without the fix. An example that passes under both the broken and the fixed code proves nothing regardless of the comments attached to it, and agents copy examples, not comments.
- **Provenance:** [1552](../../../../cases/agent-browser/1552-getbyrole-implicit-roles-regression.md) (secondary lesson).

### Choice audit (candidate, needs a decision trail to fire)

- **Trigger:** the writer emitted a decision trail (`.decisions.tsv` from the [trail-decisions](../../trail-decisions/SKILL.md) skill). Absent a trail this lens does not run — it audits declared choices, and with none declared there is nothing to audit.
- **Pass question:** each declared decision is questioned by its consequence, not by whether the diff looks right. For every row: under what runtime, input, or environment does this choice — which compiles and passes tests — produce the wrong result? The target is the decision that reads as reasonable in the diff and is bad in a context the diff cannot show (a hash that collides with a reserved range, a fix that clears state a caller still needs, an assumption about an external process). The diff-visible decisions are already covered by other lenses; the value here is the choice whose failure mode lives off-diff, stated in a row that a code-only review would never surface.
- **Why a trail and not just the diff:** most bad-decision bugs are visible in the diff in principle and get caught by the structural lenses. This lens earns its place only on the decisions with no diff footprint — "assumed the daemon does not restart", "doubled the buffer to clear the failing case" — which exist as a row and nowhere in the code. Measured caveat: in the agent-browser miss sample, roughly four of ten all-miss bugs were declared-decision-was-bad, but three of those four were also diff-visible; the lens's unique yield is the residual, so keep it gated on the trail existing and do not run it as always-on directive content (that is the terrain where narrowing hurt in the format-terrain eval).
- **Provenance:** partial, awaiting a recorded miss this lens caught that a trail-less review did not. Origin: Taelin's MatMul buffer-doubling (symptom fix, passed, shipped wrong behavior, caught only by auditing the choice) plus agent-browser [#1041](../../../../cases/agent-browser/) (djb2-hash-derived daemon port, clean in the diff, blocked by Hyper-V at runtime, maintainer-confirmed by ctate). Promote when [trail-decisions](../../trail-decisions/SKILL.md) is in a real pipeline and a genuine trail produces the miss.

### Complexity budget (candidate)

- **Trigger:** the diff adds a loop, a membership check, or a lookup inside a path that runs per-element or per-candidate (name/role collection, dedup, a scan over nodes or matches).
- **Pass question:** state the complexity in terms of the input the code actually reaches at runtime, not the happy path. A `Vec::contains` / linear scan inside a loop over N candidates is O(N^2); a per-element re-walk of the tree is O(N*depth). The worst case is the failed or large query (thousands of role matches with no name hit), not the one-element demo. If the structure is order-sensitive for output, back the membership check with a set and keep the vec for order. Correctness lenses pass this code; only asking "what is N at scale, and what runs per N?" surfaces it.
- **Provenance:** agent-browser #1552 round 3 (ctate-confirmed): `names_seen: Vec<String>` deduped with `.contains()` per role match, so a failed name query over many same-role elements was O(n^2). Invisible to every correctness lens; caught only by a complexity pass.
